# REFLECTION — 片语 RedNote 期末项目反思

## 1. 总体感受：作业交了，但学会了吗？

做完「片语 RedNote」之后，我的感受有点拧巴：像“考试作弊又没完全作弊”——项目按时交付了，测试能跑，界面能点，但我经常不确定自己**到底掌握了多少**。另一方面，Superpowers 又像一个会写代码的产品经理，帮我把从 **brainstorming → SPEC → PLAN → 拆 task 实现** 这一整套流程理得很顺；真正做决策的仍然是我——产品方向和细节取舍。

用一句话总结协作体验：**AI 负责把事情做出来，我负责想清楚要不要做、做到什么程度算完。**

## 2. 为什么做这个项目？

做这个工具，首先是因为我对电影有兴趣，想做一个**自己也会用**的创作助手，而不是为了应付课程硬凑一个题目。

在接触 Superpowers 之前，我试过直接在网页端用大模型写影评笔记：输入片名或随便聊几句，让它生成一篇能发小红书的内容。结果不理想——缺乏新意，AI 味很重，标题和正文都不够吸引人。那次尝试让我意识到：问题不在于模型不够聪明，而在于**缺少流程和质量控制**。一次性出稿看起来完成了，其实和平台上那些千篇一律的 AI 水文没多大区别。

所以「片语 RedNote」想解决的不是“能不能生成文字”，而是：**能不能用一套可复用的流程，稳定地产出一篇有想法、有网感、能配图的笔记。** 这也是我后来坚决把 Agent 最初理解的“RAG 知识库问答”改成“小红书影评创作全流程”的原因——我要的是能发出去的图文笔记，不是对着资料库聊天。

## 3. Superpowers 里哪些技能最有用？

### 3.1 Brainstorming：这次项目收获最大的技能

在我用过的 Superpowers 技能里，**Brainstorming 作用最大**。它不是替我想创意，而是**一次一个问题**，把模糊的想法一点点压实：双路线预览、Review Room 聊天室、按电影气质匹配评审团、配图也要审稿、历史草稿与收藏……很多功能点都是在对话里**被追问出来**的，不是我第一天就想好的。

如果没有这一轮，我很可能做一个“输入片名 → 出一篇字”的简陋工具——那和我之前讨厌的网页端 AI 没有本质区别。Brainstorming 还主动帮我**控制范围**：比如 80 种人格评审团不能每篇全员出场、不能爬小红书学 prompt、审稿不能无限循环等。这些提醒避免了项目在中途失控。

相比之下，一些流程性技能（比如严格按 task 派发、两阶段 code review），在单人 + Cursor 的场景里，有时会感觉**形式大于实质**——下面 TDD 那节还会提到。但 Brainstorming 和 writing-plans 产出的 SPEC/PLAN，确实是后续十多天的“轨道”。

### 3.2 writing-plans 与 subagent 推进

`writing-plans` 把 SPEC 拆成 T1 到 T15，我习惯用“开始 T9”这种方式推进。好处是**范围可控**，一次不会改崩整个仓库；缺点是**有点碎**，像是在反复催进度。回头看，对这种规模的项目，**颗粒度再大一点**可能更顺手——比如把相邻的后端 API 或前端页面合并成更少的几次派发。

我的实际验收方式也比较“产品化”：**基本不细看代码**，主要看 `pytest` / `npm test` 是否全绿，以及页面上的功能能不能点通。对小型个人项目，我的感觉是：**实现层 AI 已经能做得很好**；我更应该把精力放在前期想清楚、后期对照规约验收上。

## 4. TDD：纪律有用，但我没有真正“学会测试”

课程要求 TDD（红 → 绿 → 重构），本项目也基本照做：T1 先写一个失败的 health 测试，T15 集成测试红了再修 mock。但老实说，我的感受是：

- **拖慢速度**：Agent 往往一次就能写出能跑的实现，流程却要求先失败一次，多绕一圈。
- **形式大于实质**：很多测试是 Agent 写的，我**没有细看**，经常是跑完测试绿了就认为这个 task 过了。

换句话说，TDD 在这里更像**流程纪律**和“完成”的信号，而不是我自己练出来的测试能力。但它仍然有一个价值：当 T15 集成测试失败时，至少有一个客观结果告诉我“还没完”，而不是凭感觉宣布完成。如果再重做一次，我未必会砍掉 TDD，但会要求自己**至少读懂每个 task 的关键测试在验证什么**——那才是真正的“验收标准落地”。

## 5. SPEC、PLAN 与冷启动：文档质量决定会不会跑偏

有一件事我印象很深：**P0-B 冷启动**。用 Codex 只喂 SPEC + PLAN，它在 PLAN 门禁处停了下来，发现“P0-B 试做 T1”和“P0-B 完成前禁止 T1”自相矛盾。那一刻我体会到课程里说的：**你和主 Agent 共享的隐性上下文，会掩盖 spec 中不清晰的地方。**

这件事让我相信：**规约不是写完就完，而是需要被“陌生读者”检验。** 修订后的 PLAN 区分了“冷启动 Agent 可试做”和“主开发须等修订合并”，SPEC 也补上了 `/health`。虽然 Round 1 零代码，但修掉了一个关键的文档缺陷。

反过来，也有**规约写了、人没盯住，仍然差点跑偏**的例子：**T11** 做 Review Room 时，Agent 用手写的近似色，没有用 **nexu-io/open-design** 的官方 token。测试依然能绿，界面看起来也“差不多”，但**不符合课程要求**。是我对照规约才发现并纠正的。这说明：**测试绿 ≠ 做对了**；有些错误只有人能抓出来。

## 6. 我坚持的关键决策（人工干预）

回顾整个过程，我认为真正属于“我”的判断主要有这些：

1. **产品方向（A）**：从 RAG 问答改为小红书影评全流程——决定“做什么”。
2. **评审团设计（D）**：要有多元人格挑毛病，但接受按电影气质只激活 3–5 人——在创意与成本之间折中。
3. **审稿上限（F）**：文案与配图各自最多 5 轮——防止无限改稿。
4. **Open Design 合规（G）**：必须用官方设计 token，不能用近似色偷懒——决定“怎样才算交差”。

这些都不是实现细节，而是**方向、标准、纠偏**——也正是我在 Q8 里总结的自己的价值。

## 7. Open Design：明显减少了 AI 味，但仍有模板感

前端用了 **Linear 深色设计系统**（官方 `tokens.css`）之后，Dashboard 和 Review Room 比让 Cursor 自由发挥 Tailwind **像样很多**，AI 的廉价感弱了不少，功能布局也更容易看懂。

但用久了还是有点**模板化**：统一、专业，却不太像“我自己的品牌”。我不确定这是设计系统的局限，还是**我的提示词不够具体**（没有把电影、小红书、文艺气质说透）。无论如何，相比“Cursor 直接画页面”，Open Design 是正确的选择；**纠正官方 token** 之后，我对交付合规也放心了很多。

## 8. 如果重做，我会改什么？

结合上面的体会，如果再重做一遍，我会：

1. **把更多时间放在 Brainstorming 和检查实现、纠偏上**，而不是盯着 Agent 逐行写代码。
2. **更早想清楚“小红书笔记长什么样”**（图文一体、要有历史、要能审稿），减少中后期的大改。
3. **合并一部分 task**，降低“开始 Txx”带来的碎片感。
4. **强迫自己读懂关键测试与核心模块**，避免“测试绿了但我不知道验证了什么”。
5. **更早用真实 API 跑通一篇《肖申克的救赎》**，而不是只停留在 mock 集成测试。

核心态度不变：对这种规模的项目，实现可以 largely 交给 AI；人不能交给 AI 的是**方向、标准和纠偏**。

## 9. 对 Superpowers 与 AI4SE 的批判性看法

Superpowers 隐含了几个假设：**(1)** 规约可以先于代码写清楚；**(2)** TDD 与评审能防止 AI 协作失控；**(3)** task 拆得够细，subagent 就不会偏太远。在我的项目里：

- **(1) 基本成立**，但需要冷启动验证，否则 PLAN 自相矛盾会浪费一轮。
- **(2) 部分成立**——测试能挡掉集成层面的低级错误，但挡不住“不合规却跑得通”的跑偏（T11）。
- **(3) 基本成立**，但颗粒度过细会增加催办成本；我更倾向稍大一点的 task 加上更强的人工验收。

我对 AI4SE 的整体看法是：**AI 已经能胜任大量实现工作，工程师（或像我这样的非传统开发者）的价值越来越集中在三件事——定方向、定标准、纠偏。** 定方向：做什么、不做什么；定标准：什么叫好、什么叫完成；纠偏：测试绿了、看起来对了，但规约和用户痛点不对时，敢于叫停。

这也是我虽然感觉“像考试作弊又不像”，但仍然觉得项目有意义的原因：我做的不是一个“AI 代写的仓库”，而是**一次把真实痛点（网页端的 AI 水文）翻译成可执行规约、再推动 AI 按规约落地的练习**。不确定的部分（比如 pytest 怎么写、Docker 网络怎么配）可以事后补；确定的部分是：**没有人替我决定要做小红书影评、要 Review Room、要 5 轮上限。**

## 10. 结语

「片语 RedNote」对我来说，首先是一个**我自己会想用**的电影笔记工具实验；其次才是一次 AI4SE 的期末作业。Superpowers 最有价值的不是替我写完六千多行代码，而是逼我在写代码之前把**用户、痛点、验收标准**说清楚，并且在写代码之后保留**纠偏和验收**的环节。

如果 AI 能完成大部分编码，我仍然需要回答两个问题：**做什么？做对了吗？** 这次项目让我更清楚地看到：第二个问题，不能只交给测试的颜色。
